## Key Questions
1. 新窗口是否要求每个任务都自动调用 MCP，还是允许按任务类型跳过？
2. 优先优化推荐准确率、阅读链路完整性，还是接入与维护成本？
3. 文档真源、索引缓存和直接读取分别如何治理？

## Decisions Made
| Decision | Rationale |
|----------|-----------|
| 本轮先评审和设计，不直接修改产品代码 | 用户要求“再看一下如何优化”，且 brainstorming 技能要求设计先获确认 |
| 推荐 MCP 作为 Codex 主入口，CLI 降为管理/兜底 | MCP 参数结构化、调用稳定、无需 shell 变量，更适合新窗口自动使用 |
| 推荐 task-start + read-tree 混合流程 | 既保留最新文档树优势，又能把真实用户任务和后续阅读归入同一 trace |
| MCP 改为显式 task/trace 作用域，不使用进程全局状态 | 防止多窗口和并发任务串链 |
| CLI/ctx 不作为任何人类操作入口 | 用户明确开发者不会使用命令，只希望网页直观看到结果与效果 |
| 文档同步改为后台自动化或网页触发 | Agent 与开发者都不应承担手工 `ctx doc sync` 协议 |
| 产品正式统一为 MCP agent 入口 | 用户确认 Codex、Antigravity 等工具全部改用 MCP，不保留 CLI 作为正式使用方式 |
| 不以 MCP 调用量衡量价值 | 调用量只反映触发/采用，必须结合推荐质量和任务结果判断效果 |
| 不依赖人工反馈或任务成功率 | 用户只关注文档检索效率，且不会逐任务反馈任务是否有效完成 |
| 效果评价收缩为自动化检索指标 | 通过首次目标文档耗时、工具调用数、token、读取数量、绕过/兜底和重复读取衡量 |
| 进一步取消评测/反馈系统建设 | 用户只要可选阅读与决策链路，不需要成功率、人工反馈或独立评测平台 |
| 每任务至少一次轻量 MCP 决策入口 | AI 可自由不读文档，但若完全不触发系统，就无法知道它为何跳过 |
| 显式记录 skip/stop-reading 原因 | 用 Agent 自述原因帮助判断候选、文档边界、陈旧或缺失问题 |
| V1 不记录 skip/stop/read 原因 | 用户担心额外 MCP 调用和解释负担影响 Agent 执行效率，先只记录客观链路 |
| 任务中心采用两级页面 | 用户确认外层任务列表、内层独立详情展示 MCP 调用链路，不采用左右主从布局 |
| 产品移除原有 `ctx` 使用方式 | 用户明确要求不再保留 ctx 方式；需要将 Agent 协议、Usage 和 trace 链路统一为 MCP |

## Errors Encountered
| Error | Attempt | Resolution |
|-------|---------|------------|
| 沙箱无法访问 Docker daemon socket | 1 | 申请受控 Docker Compose 权限后成功运行测试 |
| 更新评审记录时 patch 上下文与空表格不匹配 | 1 | 读取当前文件后按精确上下文重试 |
| 原型服务器在沙箱内绑定 localhost 被拒绝（EPERM） | 1 | 经受控授权启动仅本机原型服务器 |
| Git 暂存因 `.git/index.lock` 权限失败 | 1 | 使用受控 Git 写权限，仅操作正式设计文档 |
