# 项目改进建议

本文档记录当前项目适合继续完善的方向。它不是开发记录，也不是任务上下文索引；只有在规划新能力、拆任务或评估优先级时再读取。

## 当前判断

这个项目的核心价值不是“帮 AI 读文档”，而是“帮其他项目用可观测的方式准备任务上下文”。

推荐的产品边界是：

- AI 浏览业务项目源码时，不强制经过本项目。
- AI 阅读被纳入项目管理的上下文文档时，应优先按文档树执行 `ctx read <doc-id>`。
- 根目录索引文档只负责告诉 AI 调用哪个入口命令，不承载大量知识。
- 后台负责检索、排序、trace、权限、反馈和文档管理。
- 当前阶段继续保持无向量数据库、无 embedding、无文档切分。

## 优先改进项

### P0：补齐任务入口路由

目标：让其他项目只需要维护一个很短的入口索引，AI 一上来就能执行正确命令。

建议：

- 给 `ctx prepare`、MCP `prepare_task_context` 和后端 `/api/context/prepare` 增加 `area` 参数。
- 支持记录入口来源，例如 `entrypoint_path`、`entrypoint_rule` 或 `route_hint`。
- 让 `docs/AI_CONTEXT_INDEX.template.md` 支持按 area 路由，例如支付、构建、前端、数据库等。
- `prepare` 返回中展示本次匹配的 area、入口规则和 trace_id。

价值：

- AI 不需要自己读一堆文档再筛选。
- 后台能知道它是从哪个入口、哪个任务、哪个 area 进入的。
- 后续可以统计哪些路由规则有效，哪些规则经常返回无关文档。

### P0：强化受管文档读取观测

目标：只要 AI 读取本项目管理的上下文文档，就尽量留下 trace。

建议：

- CLI `ctx read` 只要求 `doc-id`；trace 由 CLI/API 内部自动串联，不要求 AI 手动传 traceId 或 reason。
- 如果保留无 trace 读取能力，建议只作为管理端或调试端接口，并明确标记为 untracked read。
- `read` 事件中记录调用来源，例如 CLI、MCP、Web UI、API。
- 文档详情页区分“被返回过但未读取”和“已被读取”。

价值：

- 可以知道哪些文档真的被 AI 用到了。
- 可以发现 prepare 返回过多、返回不准、AI 跳读等问题。
- 便于后续优化文档质量和路由规则。

### P0：完善 trace 分析视图

目标：把本项目从“记录日志”提升到“能看懂 AI 上下文使用过程”。

建议：

- trace 详情页展示：任务文本、project、area、入口规则、返回文档、读取文档、未读取文档、反馈。
- 标记 returned but unread、read without returned、stale、missing 等状态。
- 增加 trace 过滤：项目、area、反馈类型、时间范围、是否有 missing。
- 在 dashboard 上展示最近失败路由、低质量反馈、常被读文档。

价值：

- 用户可以从后台直观看到 Codex、Claude Code 等工具的上下文使用过程。
- 后续排查“为什么 AI 没看该看的文档”会更快。

### P1：增强非向量检索质量

目标：在不引入向量数据库的前提下，提高文档命中质量。

建议：

- 用 `area` 作为强过滤或强加权项，而不是只依赖任务关键词命中。
- 给文档增加 `priority`、`freshness`、`audience`、`required_when` 等轻量元数据。
- 支持 title、doc_type、area、tags、正文命中的分项分数，方便解释为什么返回它。
- 对中文任务增加更稳的关键词提取策略，例如保留技术词、文件名、命令、接口路径。
- 对 active、stale、archived 状态设置不同权重。

价值：

- 继续保持系统简单可控。
- 返回理由更可解释，用户更容易修正文档元数据。

### P1：建立项目接入流程

目标：让其他项目低成本接入这个上下文路由系统。

建议：

- 提供命令生成项目入口索引，例如 `ctx project init-index --project app`。
- 提供项目文档导入命令，批量登记 `docs/**/*.md`。
- 给每个项目生成一份最小 `AI_CONTEXT_INDEX.md`，只写命令入口和少量规则。
- 支持项目级配置文件，例如默认 project slug、默认 area 列表、忽略路径。

价值：

- 其他项目可以沿用同一套思路，但不会被迫复制本项目的文档结构。
- 降低每次接入新仓库的人工成本。

### P1：闭环反馈与文档治理

目标：让反馈不仅被记录，还能推动文档维护。

建议：

- 将 `missing` 反馈转成待补文档建议。
- 将多次 `stale` 的文档标为待更新。
- 将长期 returned but unread 的文档标为低优先级或路由过宽。
- 增加文档健康度：最后更新时间、最近读取次数、最近反馈、是否过期。

价值：

- 文档不会只增不管。
- 后台可以帮助用户发现“上下文污染”和“过期上下文”。

### P2：MCP 和多工具适配

目标：在 CLI 稳定后，让支持 MCP 的 AI 工具更顺滑地接入。

建议：

- MCP 工具参数与 CLI 保持一致，尤其是 project、area、task、trace、reason。
- 支持 agent_name 或 tool_name，区分 Codex、Claude Code、Cursor 等来源。
- 保留 CLI 作为第一入口，MCP 作为第二阶段增强。

价值：

- CLI 更通用，MCP 更自动化。
- 后台可以比较不同 AI 工具的上下文使用行为。

## 建议的 MVP 开发顺序

1. `ctx prepare` 增加 `--area`，后端和 MCP 同步支持。
2. trace 记录 `area`、入口索引来源和调用来源。
3. 后端受管文档读取默认自动关联当前 trace；没有当前 trace 时创建直接读取记录。
4. 改造 `AI_CONTEXT_INDEX.template.md`，让它能生成按 area 路由的短索引。
5. 前端 trace 详情页增加“返回但未读取”“读取状态/来源”“反馈状态”视图。
6. 增加项目接入命令，自动生成入口索引和基础配置。

## 需要避免的方向

- 不要把大量业务知识写进根目录索引文档。
- 不要让 AI 在入口文档里手工判断应该读 A/B/C/D 多份细节文档。
- 不要为了检索效果过早重新引入向量数据库。
- 不要强制拦截 AI 对源码的正常阅读行为。
- 不要把普通聊天、状态同步、临时想法都记录进开发细节。

## MVP 验收标准

- 其他项目根目录只需要一个很短的上下文入口文档。
- AI 执行 `ctx read <doc_id>` 后，后台能看到实际读取了哪份文档。
- 文档树无法覆盖任务时，AI 再执行 `ctx prepare --project app --area payments` 获取兜底候选文档。
- 后台能看到某次任务返回了哪些文档、实际读取了哪些文档、哪些返回后没有被读取。
- 用户能根据 trace 判断是文档缺失、路由不准、文档过期，还是 AI 没有继续读取。
